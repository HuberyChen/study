Purpose of this document

These Release Notes provide information on the latest Service Release (SR) of the Vertex O Series platform. They are not meant to replace the documentation and online help. For detailed information on installing and using Vertex O Series, refer to the documentation.

For each major O Series release (for example, O Series 6.0), Vertex issues Service Releases (for example, O Series 6.0 SR1) once or twice a year that contain enhancements and fixes for defects. In between Service Releases, Vertex issues Maintenance Packs (for example, O Series 6.0 SR6.1) that address higher-priority defects.

All information in these Release Notes and the accompanying documentation is subject to change with subsequent service releases. Before attempting to install Vertex O Series, visit the myVertex Customer Portal (http://my.vertexinc.com) and download the latest service release to ensure that you have the most up-to-date software, Release Notes, and documentation.

Changes in 6.0 SR7.6

Calculation Engine changes

  1. Distribute tax transactions that force a tax on a freight amount are now supported. This change fixes an issue with Distribute Tax transactions, where a rounding difference between the original single Invoice transaction and the corresponding multiple Distribute Tax transactions caused the Distribute Tax transactions to fail. To force a tax in a Distribute Tax transaction on a freight amount of $0.00, set the following parameters:
    • tps.common.domain.force_distributetax_on_failure.supplies=true
    • tps.common.domain.force_distributetax_on_failure.procurement=true
  2. Invoice Verification events that invoke user-defined basis rules now correctly return sales tax. This change fixes an issue where these events were incorrectly returning consumer use tax after invoking user-defined basis rules.
  3. Processing for Brazil transactions now accommodates reduction of the Substitution Tax by a specified imposition and/or a specified percentage. The following two new configuration parameters support this change:
    • com.vertexinc.tps.common.domain.substitution.credit.imposition=xxx;y (where xxx is the Imposition Type ID and Y is the ImpositionType source ID)
    • com.vertexinc.tps.common.domain.substitution.credit.reduction.flexfield=SUBCRED (where SUBCRED should be replaced by the name of the numeric flex field that holds the ICMS credit percentage)

    O Series automatically deducts the Substitution Tax by 100% of the ICMS Reseller Tax.  The new configuration parameters provide flexibility by enabling you to deduct the Substitution Tax by an additional imposition or by a specified percentage of the ICMS Reseller Tax or other imposition.

    With the imposition parameter, you can specify a Vertex-defined imposition or a user-defined imposition that will be deducted from the Substitution Tax, along with the ICMS Reseller Tax.

    With the flexfield parameter, you can change the percentage of the ICMS Reseller tax or other imposition that is deducted from the initial Substitution Tax amount. To use this parameter, set up a numeric flexible field that specifies the percentage of the ICMS Reseller Tax that is subtracted from the Substitution Tax. To apply the parameter to multiple jurisdictions, also set up a Lookup Table by NCM and Destination State to derive the ICMS credit reduction percentage, as well as a Tax Assist rule that populates the flexible field value with the ICMS credit percentage.

    Depending on your objective, use these parameters either individually or in combination, as follows:

    • To reduce the Substitution Tax by a specified percentage of the ICMS Reseller tax, use only the flexfield parameter.
    • To reduce the Substitution Tax by 100% of both the ICMS Reseller tax and another imposition, use only the imposition parameter.
    • To reduce the Substitution Tax by a specified percentage of both the ICMS Reseller tax and another imposition, use both parameters.  (Note that this scenario could result in a double deduction of the ICMS Reseller tax if both taxes are calculated for a given transaction.)
  4. Sales tax holiday rules in the U.S. state of Georgia can now be overridden correctly by user-defined rules.
  5. An issue with bracket tax calculation for the U.S. state of Maine, where O Series calculated an incorrect tax amount for a transaction having a basis rule and multiple line items, has been resolved.
  6. An issue with U.S. state of Pennsylvania bracket tax calculations, where invoice tax amounts were inaccurate for transactions that included user-defined rules apportioned by taxable amount, has been resolved.
  7. Effective rates in requests with multiple line items are now rounded to 6 decimal places.

Changes in 6.0 SR7

Calculation Engine changes

  1. For O Series setups where some customers and/or vendors have the same codes as taxpayers, the Calculation Engine now uses only the taxpayer registration information to determine taxability. This resolves an issue where the Calculation Engine incorrectly used the customer or vendor registration information to determine taxability.
  2. An invoice-level, U.S. Max Tax tier rule now processes correctly when the first tier is nontaxable/exempt, the second tier is taxable, and the Tax all at top level option is turned off. The rate is now applied separately for each tier.
  3. Accrual transactions, in which both the parent line item and child line item have a vendor-charged tax, are no longer adding the child line item's tax to the parent line item's tax.
  4. An issue where imposition-specific exempt amount overrides were not processed has been resolved. If the amount associated with the override is 100 percent of the taxable basis, the tax is also completely exempt. If the amount associated with the override is partially taxable, then flat fees and quantity-based taxes fully apply.
  5. An issue where a higher-level jurisdiction Basis rule overrode a lower-level jurisdiction Basis rule has been resolved. A higher-level jurisdiction Basis rule is now only applied to a lower-level jurisdiction if the lower level does not have its own Basis rule.  A user-defined lower-level rule takes precedence over a Vertex-supplied lower-level rule.
  6. An issue where tax was not calculated correctly on a tax-inclusive line item having an embedded freight amount has been resolved.
  7. For an apportioned, tax-inclusive freight line item, the derived extended amount plus the tax amount now match the input total price.
  8. In Distribute Tax transactions, the Calculation Engine now correctly calculates the filing tax amount by multiplying the input tax amount by the filing currency conversion rate. Previously, O Series calculated the filing tax amount in the filing currency, but did not include the input tax amount in the calculation.
  9. For transactions that involve filing currency conversion and an allocation rule, the issue that caused a null pointer exception error has been resolved.
  10. The Calculation Engine now returns correct tax percentages and amounts for the U.S. state of Alabama sales tax holiday county-level rules that have conditional jurisdictions and defer to state-level tiered rules with taxable tiers that use the standard rate.
  11. An issue where nontaxable transactions with a price of $0 in the U.S. state of Texas were failing has been resolved.
  12. An issue where a nontaxable rule in the U.S. state of Iowa returned a taxable result for a $0 extended price has been resolved.
  13. To reduce the possibility of overcollecting Value-Added Tax (VAT) because of rounding, rounding adjustments are now made at the invoice level to account for tax-inclusive line items.
  14. The correct tax registrations are now returned consistently in the calculation response when both country-level input tax registrations and state-level input tax registrations are included in a calculation request.
  15. An issue for zero-rated, taxable, Intra-EU transactions, where there is no Buyer Registration ID in the output, has been resolved. For Intra-EU transactions where the TaxResultType is Taxable, the Output Notice specifies Intra-EU, Zero Rate, and No Tax Responsibility conclusions, and there is no Buyer Registration ID in the Destination country, the Calculation Engine sets the Buyer Registration ID to the first Customer Registration ID that it finds when searching alphabetically by EU country name.
  16. For Input Value-Added Tax (VAT) transactions where a taxpayer-level recoverable percent is set up, the Calculation Engine now allows a transactional recoverable percent override at the line item level. This logic applies both to Purchase Order events where Calculate Vendor Tax is selected and to Evaluated Receipt Settlement (ERS) events where Calculate Self-Accrual is selected. Previously, the recoverable percent was incorrectly set to 0 (zero) in the XML response message when there was both a taxpayer-level recoverable percent and a transaction-level recoverable percent in a Purchase Order event or ERS event.
  17. Recoverable Percent values are now rounded to six decimal places.
  18. Filing Currency Conversion Rates are now rounded to six decimal places.
  19. Transactions that include tax calculations from both the buyer and seller perspective are now rounding amounts correctly based on the currency unit for the transaction. For example, Purchase Order amounts that are reported in Hungarian currency (HUF) are now truncated to a whole number. Previously, these amounts included decimal values.
  20. An issue with a leasing calculation involving more than one situs location, where a district tax was calculated based on the extended amount rather than asset cost, has been resolved. The calculation logic has been modified to apply the parent jurisdiction tax basis rule to all lower jurisdictions when there are multiple situs locations.

Known issues

Installation

  • DB2 tablespace names must not contain a dash. DB2 jdbc drivers will not correctly parse database table create statements with a dash in the tablespace name, and the Vertex O Series installation will fail.

Servlet container

  • Vertex recommends using the JAXP 1.2 XML parser from Sun, which is provided with the Vertex Web applications. If your servlet container follows section SRV.9.7.2 of the Java Servlet Specification, then it uses JAXP 1.2. Note that some servlet containers may intercept the JAXP 1.2 XML parser. If this is the case with your servlet container, refer to the documentation for your servlet container for details on configuring the Vertex-supplied XML parser so that it functions properly in this environment. Also, refer to the Installation Guide for information on preparing the servlet container.

Integrations

  • If you are planning to perform transactions that involve the conversion of one currency to another you must set up the interface between your Enterprise Resource Planning (ERP) software and Vertex O Series to pass Vertex conversion rates and currency codes for the locations in the transaction. The locations may be any or all of the following that are applicable in your situation:
    • Physical Origin (PO)
    • Administrative Origin (AO)
    • Administrative Destination (AD)
    • Destination (D)
  • For performance reasons, objects returned by the API may reference objects used internally by the system. Mutable objects returned by the API should not be modified because modifying internal attributes can cause unpredictable results. Examples of mutable objects returned by the API are java.util.Date and Java arrays.

WebLogic

The following issues exist in deployments that use selected versions of WebLogic:
  • In deployments that use WebLogic 10.3.3, 10.3.4, 10.3.5, or 10.3.6 with AIX 6 or 7 and Address Cleansing, the normal EAR deployment for O Series SR4 and later does not work properly.     1. Edit the WAR file to remove the metro jar files webservices*.jar.

    3. Repackage and deploy the WAR file.
  • In deployments that use WebLogic 10.3.6 with Windows, the normal EAR deployment for O Series SR4 and later does not work properly. There is no workaround. WebLogic 10.3.5 and earlier work with Windows.

Installation notes

If you are installing Vertex O Series for the first time, refer to the instructions in the Installation Guide to install Vertex O Series 6.0, and then follow the instructions below to apply this Service Release. Applying this Service Release installs files on your system. Do not remove these files.

Vertex O Series Service Releases are cumulative. For example, if you are on O Series v6.0 and apply the v6.0 SR3 patch, you will install all the changes made in v6.0 SR1 through v6.0 SR3. Should the latest Service Release installation fail, Vertex Support can help you revert easily to your previous version. Do not install several Service Releases one after the other at one time or you will have to back out each Service Release separately to revert to your previous version

Applying 6.0 SR7.7

Before you begin

  • If you have multiple instances of O Series pointing to one database and are applying a patch you must wait until the patch application is completed before applying the patch to the next instance. For example, you may have multiple instances of O Series for load balancing.
  • Review the Installation Guide chapter titled "Deploying Your Servlet Container" as well as the pre- and post- installation chapters for any notes or special instructions related to your particular Web application server.
  • If you plan to deploy O Series using Oracle WebLogic, you must deploy the O Series .ear files, not the .war files.
  • Ensure you have the correct Vertex O Series License file, and that it is copied into the Vertex Home directory.
  • Ensure that the Web Archive (WAR) and Enterprise Archive (EAR) files for the O Series components are in your Web directory. If these files were moved, the service release may not install correctly.
  • Ensure that your application server user account and the user account used to install this service release have full read, write, create, and delete access to the Vertex Home directory and all subdirectories.
  • If you have purchased Vertex Reporting and Analysis, ensure that your database user ID has permission to create a database view.
  • Vertex recommends that you turn off schema validation before applying a Service Release.
  • Vertex recommends that you back up all O Series databases before applying a Service Release. To ensure the integrity of the system, you should always back up all O Series databases at the same time. If you need to restore any one database, you must restore all the databases.
  • If you are applying this Service Release to a system running the AIX 6.1 Operating System, be sure to perform the installation of the Service Release as a user in the root group.

To apply this Service Release

  1. Copy the JAR file for the service release (vertex-o-series-patch-6.0.7.7.1.jar) to the patch directory located under the Vertex Home directory, which is the root directory in which TPS is installed.
  2. Change the working directory to the patch directory.
  3. Stop the application server.
  4. Undeploy the Web applications.

  5. Enter the following command to launch the installer and apply the service release:

    java -jar -Djdbc.classpath=jdbc_driver_files vertex-o-series-patch-6.0.7.7.1.jar

    where:

    jdbc_driver_files is the full path and file name of the installed jdbc driver file or files. This is only necessary when using a jdbc driver other than one supplied by Vertex.
  6. Deploy the Web applications. Refer to the Installation Guide for details.

Special instructions for IBM DB2 UDB for iSeries V5R4 with i5/OS users

If you are using IBM DB2 UDB for iSeries V5R4 with i5/OS, complete the following steps to install the service release:
  1. Start Qshell.

  2. Enter the following command to launch the installer and apply the service release:

    java -jar -Djdbc.classpath=/drvpath/jt400.jar/path/vertex-o-series-patch-6.0.7.7.1.jar

    where:

    drvpath is the complete file path to the jt400.jar file

    path is the complete file path to the Installation file

Removing 6.0 SR7.7

Before you begin

Back up the information in your vertex.cfg file before removing the patch. When you remove the patch, a new Vertex configuration file (vertex.cfg) will be created and the current vertex.cfg file will be overwritten, causing any information you entered to be lost. Caution: Do not replace the vertex.cfg created during patch removal with the vertex.cfg from the current patch. Doing so could result in errors or undesired system behavior because configuration options may have changed.

To remove this Service Release

  1. Stop the application server.
  2. Undeploy the Web applications.
  3. Change the working directory to the Vertex Home\patch directory.

  4. Enter the following command to launch the installer and remove the service release:

    java -jar -Djdbc.classpath=jdbc_driver_files vertex-o-series-patch-6.0.7.7.1.jar -remove

    where:

    jdbc_driver_files is the full path and file name of the installed jdbc driver file or files. This is only necessary when using a jdbc driver other than one supplied by Vertex.

  5. Deploy the Web applications. Refer to the Installation Guide for details.
